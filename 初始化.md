你的迁移方案已经非常成熟了，整体方向基本是当前 Windows 高保真播放器里“长期可维护性 + 性能 + 工程体验”最合理的一条路线。尤其是：

- 放弃 Qt/QML 与 C++ 历史包袱
- 彻底 Rust 化底层
- Tauri 做系统壳层
- React 做 UI
- WASAPI Exclusive + DSD 保留

这个架构在未来 3~5 年都不会过时。

下面我重点说“还能进一步优化的地方”和你提到的三个 Open Questions。

------

# 一、整体架构还能优化的地方

------

## 1. 不建议让 Tauri Backend 直接承载整个 Audio Engine

你现在写的是：

> Rust 引擎封装到 Tauri State 中

这个在早期没问题，但后期会出现：

- 生命周期混乱
- IPC 与音频线程耦合
- 热重载问题
- Tauri 升级影响核心引擎
- 单元测试困难

更推荐：

```text
Frontend (React)
        ↓ IPC
Tauri Shell Layer
        ↓
seraph-core (Rust crate)
        ↓
audio-engine
```

即：

```text
src-tauri/
 ├── src/
 │    ├── main.rs          <- Tauri 壳
 │    └── ipc/
 │
crates/
 ├── seraph-core/
 ├── seraph-audio/
 ├── seraph-playlist/
 ├── seraph-dsd/
 ├── seraph-resampler/
 └── seraph-visualizer/
```

这样未来：

- CLI 版本
- Headless 版本
- Linux 版本
- macOS CoreAudio
- WebSocket Remote
- 插件系统

都能扩展。

这是非常关键的一步。

------

# 2. 不建议一开始就直接用 symphonia 全替代 FFmpeg

这是一个很容易踩坑的地方。

`symphonia` 很优秀，但目前：

| 格式     | 状态     |
| -------- | -------- |
| FLAC     | 很稳定   |
| MP3      | 稳定     |
| WAV      | 稳定     |
| AAC      | 一般     |
| ALAC     | 一般     |
| Opus     | 部分支持 |
| CUE      | 没有     |
| SACD ISO | 没有     |
| Ape      | 不成熟   |

对于 HiFi 用户：

- CUE
- ISO
- TAK
- APE
- WV

很容易成为刚需。

## 更推荐：

### 第一阶段：

```text
symphonia + ffmpeg-next
```

策略：

```rust
if symphonia_supports() {
    use_symphonia();
} else {
    fallback_ffmpeg();
}
```

后期再逐步去 FFmpeg 化。

否则你会在：

- 边缘编码
- VBR Seek
- metadata
- 奇怪采样率

上花大量时间。

------

# 3. 频谱/VU 不建议走 IPC 高频推送

你现在写：

> emit 推送频谱分析数据

如果以后：

- 60 FPS 频谱
- FFT 2048
- 左右声道
- 峰值保持

IPC 会成为瓶颈。

更好的方案：

------

## 推荐结构

Rust：

```text
Shared Memory / RingBuffer
```

React：

```text
requestAnimationFrame()
```

前端主动拉。

类似：

```text
Rust FFT Thread
    ↓
Shared Atomic Buffer
    ↓
Frontend Polling
```

这样：

- 不阻塞音频线程
- 不阻塞 Tauri IPC
- 延迟更低

这是专业播放器常见方案。

------

# 4. 你的 RingBuffer 可以进一步升级

你现在写：

> crossbeam SPSC

没问题。

但对于真正 Pro Audio：

推荐：

## `ringbuf`

或者：

## `rtrb`

比 crossbeam 更适合音频。

原因：

| crate     | 特点          |
| --------- | ------------- |
| crossbeam | 通用并发      |
| ringbuf   | 音频友好      |
| rtrb      | realtime-safe |

尤其：

```rust
rtrb
```

是专门为实时音频设计。

非常适合：

```text
Decoder Thread -> Audio Callback
```

------

# 5. 建议提前抽象 Audio Backend Trait

即使现在只做 WASAPI。

不要直接：

```rust
WasapiPlayer
```

而是：

```rust
trait AudioBackend {
    fn play(...)
    fn pause(...)
    fn submit(...)
}
```

实现：

```rust
WasapiExclusive
WasapiShared
AsioBackend
CoreAudioBackend
AlsaBackend
```

否则未来 Linux/macOS 会非常痛苦。

------

# 6. 建议不要让 React 直接管理“播放状态机”

播放器真正的状态机：

```text
Stopped
Loading
Buffering
Playing
Paused
Seeking
Transitioning
DeviceLost
```

应该完全在 Rust。

React 只做：

```text
UI Projection Layer
```

否则：

- race condition
- 双向状态同步
- UI 不一致

会越来越严重。

------

# 7. 建议尽早建立“消息总线”

你现在：

```text
invoke / emit
```

后期会爆炸。

推荐：

```rust
enum PlayerEvent {
    PlaybackStarted,
    TrackChanged,
    DeviceLost,
    Progress,
    Spectrum,
}
```

统一 EventBus。

React：

```ts
usePlayerEvents()
```

这是后期维护关键。


- minimum phase

这样会非常专业。

# 三、还有几个你现在没写但非常重要的点

------

# 1. Exclusive Mode 的 Device Lost

这是最容易被忽视的。

比如：

- DAC 被拔掉
- 采样率变化
- 默认设备切换
- 睡眠恢复

一定要有：

```text
Device Recovery State Machine
```

否则播放器会随机炸。

------

# 2. DSD 不建议直接绑定 DoP

未来最好支持：

| 模式           | 说明     |
| -------------- | -------- |
| Native DSD     | ASIO     |
| DoP            | WASAPI   |
| PCM Conversion | fallback |

否则以后扩展困难。

------

# 3. Gapless 真正难的是：

## clock continuity

不是预读取。

你需要：

```text
sample-accurate transition
```

尤其：

- FLAC album
- live concert

否则会有 click/pop。

------

# 4. 建议尽早建立日志系统

推荐：

```rust
tracing
tracing-subscriber
```

不要简单 println。

音频问题极难调试。

------

# 5. 建议一开始就做 Benchmark

比如：

- callback underrun
- decode latency
- ringbuffer occupancy
- resampler cost

否则后期很难定位爆音问题。

------

```text
Seraph
```
