## 三、必须优先修复的问题(TOP 10)

### 🔴 1.【P0】B 站登录 Cookie(SESSDATA)可泄漏到任意 URL —— 账号接管风险
`src-tauri/src/ipc/bilibili/import_audio.rs:50-56`:导入框输入任意 `https://` 链接(不含 BV 号)时,直接用**携带登录 Cookie 的 client**(`session.rs:6-16` 将 Cookie 放入 default_headers)向该 URL 发 GET。reqwest 仅在跨域重定向时剥离 Cookie,首个请求不剥离。攻击者诱导用户粘贴伪装"B 站分享链接" → SESSDATA 直达攻击者服务器。
**修复**:解析短链/网页改用无 Cookie 裸 client + host 白名单(`b23.tv`/`acg.tv`/`*.bilibili.com`)。详见 04 报告 P0-1。

### 🔴 2.【P0】曲库缓存非原子写 + 读损坏静默当空库 —— 整个曲库可静默蒸发
`media_library.rs:66-68`(`fs::write` 直接覆盖数 MB 的 library-cache.json)+ `media_library.rs:78`、`commands.rs:64`(`read_cached_tracks().unwrap_or_default()`)。写一半断电/崩溃 → JSON 截断 → 下次导入把损坏当空库,**用只含本次导入的列表覆盖整个文件**,全部曲目与手动匹配的歌词丢失。
**修复**:temp+rename 原子写;读损坏时报错并备份坏文件,而非当空库。详见 04 报告 P0-2。

### 🟠 3.【P1】全引擎无增益 ramp —— 暂停/恢复/seek/音量变化均硬切,有爆音风险
`engine.rs:1422-1451`、`wasapi.rs:398-403`。HiFi 播放器核心听感问题。修复集中在 `render_output`:每样本 5ms 增益斜坡;暂停先 ramp 后静音;seek 先 ramp-out 再切 generation。详见 02 报告 F-1。

### 🟠 4.【P1】DSF 末尾 padding 解码为满幅负 DC —— 每首 DSF 结尾"咔/砰"
`dsd.rs:295/298-301/146-209`:`sample_count` 只用于算时长,块对齐 padding(bit=0 → −1.0)被当音频解码,DSD64 最长 11.6ms 满幅直流。修复:按 `sample_count.div_ceil(8)` 截断解码。详见 03 报告 F-1。

### 🟠 5.【P1】DC blocker 截止实为 35Hz(注释称 7Hz)且随 DSD 倍率翻倍至 140Hz —— 系统性低频削薄
`dsd.rs:21/550-563`。按实际 PCM 率计算 R(目标 ~2Hz)。详见 03 报告 F-2。

### 🟠 6.【P1】DSD 容器头无上限校验 —— 一个损坏的 .dsf/.dff 扫库即可让整个应用 OOM abort 或死循环
`dsd.rs:286/351/293-296` 巨型分配、u64 溢出回绕死循环。修复:字段合理性校验 + chunk 限额 + checked_add 单调推进。详见 03 报告 F-4。

### 🟠 7.【P1】5.1/7.1 → 立体声直接丢弃中置/环绕声道 —— 多声道音源人声消失
`engine.rs:1518-1544`。实现 ITU 标准下混。详见 03 报告 F-3 / 02 报告 F-17(同一问题)。

### 🟠 8.【P1】B 站音频下载复用 30 秒总超时 client —— 大文件/慢网导入必然失败
`session.rs:7-9` 的 `Client::timeout` 是整个请求周期总超时,下载数百 MB FLAC 必超时,备用 URL 重试同样失败。修复:下载专用 client 用 `connect_timeout` + `read_timeout`,不设总超时。详见 04 报告 P1-1。

### 🟠 9.【P1】前端"记住上次播放曲目"完全失效 —— 每次启动重置到第 1 首
`useHydratePlayerStore.ts:16-17` + `libraryActions.ts:333-344/231-246`:持久化的是索引而 playlist 不持久化,水合时 `min(index, 0)` = 0。修复:持久化曲目 id 并在加载后按 id 重映射。详见 05 报告发现 1。

### 🟠 10.【P1】快速切歌无请求代际守卫 —— 慢速重缓存的旧曲目会顶掉新选中的曲目
`playbackActions.ts:232-271` + `bilibiliActions.ts:7-31`:点击需重新下载的曲目 A(数秒)后再点 B,A 完成后其挂起的 `sendPlayCommand(A)` 把 B 顶掉。修复:模块级 playEpoch 代际计数。详见 05 报告发现 2。

## 七、审查确认无问题的方面(四通道一致结论)

- 音频回调实时安全(无锁/无分配/无 IO,除 seek 清扫一处)、Atomic ordering 正确、rtrb 生产消费逻辑正确
- 子进程管理(kill+wait、Drop 兜底、stderr 限额、无 shell 注入)
- 凭据存储(Cookie 已入 Windows Credential Manager,session 文件不落 Cookie,含迁移逻辑)
- 歌词解析防御(BOM/UTF-16/GBK/畸形时间戳/zlib bomb 8MB 上限)
- 缓存 marker 机制、导入递归/junction 防护(曲库端)、持久化损坏 JSON 自清理
- 时间戳精度、DSF/DFF 交错索引、字节序处理
